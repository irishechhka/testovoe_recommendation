# Next Item Recommendation

Простая рекомендательная система, предсказывающая следующий товар на основе истории просмотров пользователя. Реализована на чистом Python без ML-библиотек — только `collections`, `json` и `matplotlib`.

---

## Задача

По последовательности просмотренных товаров в сессии предсказать, какой товар пользователь посмотрит следующим. Метрика качества — **Hit@10**: насколько часто правильный ответ попадает в топ-10 рекомендаций.

---

## Подход

Модель строит **граф переходов** между товарами: если пользователь просмотрел товар `i`, а затем товар `j`, это считается переходом `i → j`. По всем обучающим сессиям считается вероятность:

```
P(j | i) = count(i → j) / Σ count(i → *)
```

При предсказании модель берёт последний товар в сессии и возвращает топ-10 товаров с наибольшей вероятностью перехода.

**Обработка граничных случаев:**
- Самопереходы `i → i` исключены из графа (не несут информации)
- Если товар не встречался при обучении (cold-start) → fallback к топ-10 популярных
- Если соседей в графе меньше 10 → добираем из топ популярных

## Данные

Формат файла `sessions.jsonl` — каждая строка это JSON-массив с ID товаров, упорядоченных по времени просмотра:

```json
[17, 42, 8, 103, 42]
[5, 99, 203, 17]
[203, 17, 88, 42, 7, 19]
```

### Train / Test Split

Для каждой сессии: все товары кроме последнего → обучение, последний товар → тестовая цель.

```
Сессия:  [a, b, c, d]
Train:   [a, b, c]
Target:  d
```

**Почему не случайное разбиение?** Данные — это последовательности во времени. Случайное разбиение создаёт утечку данных: модель может обучиться на переходе `b → c` и потом использовать это знание, предсказывая `b` в тесте. Правильная схема — всегда обучаться на начале, предсказывать конец.

## Результаты

| Модель | Hit@10 |
|---|---|
| Бейзлайн (топ-10 популярных) | ~0.07 |
| Markov-модель (граф переходов) | ~0.15 |

Markov-модель превосходит бейзлайн примерно вдвое за счёт учёта локального контекста: зная, что пользователь смотрел конкретный товар, модель рекомендует то, куда чаще всего переходили другие пользователи из этого же товара.

---

## Ограничения и возможные улучшения

- Модель учитывает только **последний** товар, игнорируя всю предыдущую историю сессии
- Можно учесть историю целиком (например, через взвешенные переходы с затуханием)
- При большом числе холодных товаров качество падает — можно использовать эмбеддинги товаров
- Марковское предположение (следующий элемент зависит только от текущего) сильно упрощает реальное поведение пользователей
